## Storage & CDN Setup (Backblaze B2 + Cloudflare)

This guide explains how to configure Backblaze B2 buckets and connect them behind Cloudflare CDN for this project.

---

### 1. Buckets you need to create

From the storage configuration (`storage.schema.ts` and `config.service.ts`), the application expects the following canonical buckets by default (you can override names via environment variables if you prefer):

- **Private bucket**: `nestlancer-private` (`STORAGE_BUCKET_PRIVATE`, also `S3_PRIVATE_BUCKET` in `MediaConfig`)
- **Public bucket**: `nestlancer-public` (`STORAGE_BUCKET_PUBLIC`, also `S3_PUBLIC_BUCKET` in `MediaConfig`)
- **Avatars bucket**: `nestlancer-avatars` (`STORAGE_BUCKET_AVATARS`, default for `usersService.avatar.s3Bucket`)
- **Request attachments bucket**: `nestlancer-requests` (`STORAGE_BUCKET_ATTACHMENTS`, default for `requestsService.attachments.s3Bucket`)
- **Quotes PDFs bucket**: `nestlancer-quotes-pdfs` (`STORAGE_BUCKET_QUOTES`)
- **Deliverables bucket**: `nestlancer-deliverables` (`STORAGE_BUCKET_DELIVERABLES`)
- **Reports bucket**: `nestlancer-reports` (`STORAGE_BUCKET_REPORTS`)
- **Generic PDFs bucket**: `nestlancer-pdfs` (`STORAGE_BUCKET_PDFS`, hard-coded as `pdfs` in some services)

You can either:

- Create **all** of these as separate B2 buckets, or  
- Map multiple logical buckets to a **single physical B2 bucket** by setting the corresponding `STORAGE_BUCKET_*` env vars to the same B2 bucket name.

For simplicity, a common production setup is:

- One **private** bucket for user uploads and sensitive files (e.g. `nestlancer-private`)
- One **public** bucket for files that will be served via CDN (e.g. `nestlancer-public`)

#### Plan A: Two physical buckets with backup (recommended)

Use **two main buckets plus two replica buckets** for production and backup. The app still uses logical bucket names in config; you point them all to the same two physical B2 bucket names.

| Role | Physical B2 bucket name | Purpose |
|------|------------------------|---------|
| **Primary private** | `prod-private` | All private/sensitive data: media, attachments, quotes, deliverables, reports, receipts/invoices. Use path prefixes per logical type (e.g. `private/avatars/`, `private/requests/`, `private/pdfs/`). |
| **Primary public** | `prod-public` | All public assets: CDN origin, health check, public avatars if desired. |
| **Replica private** | `prod-private-backup` | Cloud Replication target for `prod-private` (backup). |
| **Replica public** | `prod-public-backup` | Cloud Replication target for `prod-public` (backup). |

- **Env mapping for Plan A:** Set every private logical bucket to the same physical bucket, and every public one to the other:
  - All of: `STORAGE_BUCKET_PRIVATE`, `STORAGE_BUCKET_ATTACHMENTS`, `STORAGE_BUCKET_QUOTES`, `STORAGE_BUCKET_DELIVERABLES`, `STORAGE_BUCKET_REPORTS`, `STORAGE_BUCKET_PDFS` (and `STORAGE_BUCKET_AVATARS` if avatars are private) â†’ **`prod-private`**.
  - `STORAGE_BUCKET_PUBLIC` (and `STORAGE_BUCKET_AVATARS` if avatars are public) â†’ **`prod-public`**.
- **Backup cost:** You pay standard B2 storage for both primary and replica buckets (2Ă— storage); replication itself has **no extra fee** and **no egress charge** between buckets.
- **Path prefixes:** If you ever need to separate data by type inside one bucket, use key prefixes (e.g. `avatars/`, `requests/`, `pdfs/`) in your upload keys; the codebase can already use different key paths per service.

### 1.1 Bucket-by-bucket permissions and CDN usage

Below is how each bucket is used in the codebase and what permissions it should have.

- **`nestlancer-private` (private bucket)**
  - **Used by**:
    - `MediaConfig.S3_PRIVATE_BUCKET` and `MediaStorageService` for user media uploads (presigned PUT/GET).
    - `MediaService` (`directUpload`) stores user media directly in this bucket.
    - `DeliverablesService` generates signed URLs from this bucket for deliverables:
      - `getSignedUrl({ bucket: 'nestlancer-private', key: 'deliverables/<mediaId>', ... })`.
  - **Access level**:
    - **Must be private** in Backblaze B2.
    - Access should happen only via **signed URLs** generated by the backend.
  - **CDN recommendation**:
    - You can still put this behind Cloudflare, but only for **signed URLs**.
    - Do **not** expose it as a public directory listing; do not make this bucket public in B2.

- **`nestlancer-public` (public bucket)**
  - **Used by**:
    - `MediaConfig.S3_PUBLIC_BUCKET` (intended for publicly accessible media or derivatives).
    - `StorageHealthService` checks connectivity via:
      - `storageService.exists('nestlancer-public', '.healthcheck')`.
  - **Access level**:
    - Should be **public** in Backblaze B2 (objects readable without auth).
    - Ideal for static assets, thumbnails, and any media that can be shared publicly.
  - **CDN recommendation**:
    - This is the **main bucket to put behind Cloudflare CDN**.
    - Point your Cloudflare `CNAME` (e.g. `cdn.yourdomain.com`) to this bucketâ€™s host.
    - Configure strong `cache-control` headers and Cloudflare cache rules (see sections below).

- **`nestlancer-avatars` (avatars bucket)**
  - **Used by**:
    - `usersService.avatar.s3Bucket` (default: `nestlancer-avatars`).
    - `AvatarService` uploads avatars and stores the full URL in the DB.
  - **Access level**:
    - Recommended to be **public** (user profile images are typically public).
    - If you need privacy, keep it private and always serve via signed URLs or via a proxy endpoint.
  - **CDN recommendation**:
    - If avatars are public: serve them via Cloudflare by:
      - Either mapping this bucket directly with another CDN subdomain (e.g. `avatars.yourdomain.com`), or
      - Storing avatars in the main public bucket and using path prefixes (e.g. `public/avatars/...`).
    - If avatars should be private, treat like `nestlancer-private` and rely on signed URLs instead.

- **`nestlancer-requests` (request attachments bucket)**
  - **Used by**:
    - `requestsService.attachments.s3Bucket` (default: `nestlancer-requests`).
    - `RequestAttachmentsService`:
      - Uploads user files to this bucket.
      - Stores the **public URL** returned from `storageService.upload` in `fileUrl`.
  - **Access level**:
    - Attachments can contain potentially sensitive client documents.
    - **Recommended** to treat this as **private** and:
      - Either: keep the bucket private and use **signed URLs** when exposing attachments.
      - Or: move attachments to `nestlancer-private` and only keep public-safe files here.
  - **CDN recommendation**:
    - If you keep it private: use signed URLs only (optionally cached by Cloudflare).
    - If you decide it can be public: add this bucket (or a path prefix) behind Cloudflare with moderate cache TTLs.

- **`nestlancer-quotes-pdfs` (quotes PDFs bucket)**
  - **Used by**:
    - Config only (`STORAGE_BUCKET_QUOTES`), typically for generated quote PDFs.
  - **Access level**:
    - These documents usually contain pricing and client info â†’ treat as **private**.
  - **CDN recommendation**:
    - Keep bucket private, serve via **signed URLs** only.
    - Cloudflare can still cache signed responses if desired, but do not make the bucket public.

- **`nestlancer-deliverables` (deliverables bucket)**
  - **Used by**:
    - Config only (`STORAGE_BUCKET_DELIVERABLES`); actual signed URLs are generated from `nestlancer-private` in `DeliverablesService` (`deliverables/<mediaId>`).
  - **Access level**:
    - Treat as **private**, deliverables are often paid work products.
  - **CDN recommendation**:
    - Same pattern as `nestlancer-private`: private bucket with **signed URLs**.

- **`nestlancer-reports` (reports bucket)**
  - **Used by**:
    - Config only (`STORAGE_BUCKET_REPORTS`), presumably for admin/analytics reports.
  - **Access level**:
    - Generally **private** (contains business metrics).
  - **CDN recommendation**:
    - Keep private, serve with **signed URLs** if downloading via web.

- **`nestlancer-pdfs` / `pdfs` (generic PDFs bucket)**
  - **Used by**:
    - `ReceiptPdfService` and `InvoicePdfService`:
      - Upload to bucket `'pdfs'`.
      - Generate signed URLs with `getSignedUrl({ bucket: 'pdfs', ... })`.
    - `STORAGE_BUCKET_PDFS` defaults to `nestlancer-pdfs`, but code currently hard-codes `'pdfs'` for receipts/invoices.
  - **Access level**:
    - Contains invoices and receipts â†’ **must be private**.
    - Always accessed via **signed URLs**, never public listing.
  - **CDN recommendation**:
    - Private bucket only; Cloudflare may cache signed downloads but do not expose the bucket publicly.

### 2. Configure Backblaze B2

1. **Enable B2 in your Backblaze account**
   - Log in to Backblaze.
   - Go to **B2 Cloud Storage** and ensure it is enabled.

2. **Create required buckets**
   - For each bucket you want as a distinct physical bucket, create it in B2 with the corresponding name (e.g. `nestlancer-private`, `nestlancer-public`, etc.).
   - For **Plan A** (recommended): create exactly **four** buckets:
     - `prod-private` (private)
     - `prod-public` (public, for CDN)
     - `prod-private-backup` (private, replication target)
     - `prod-public-backup` (public or private, replication target)
   - For CDN delivery via Cloudflare you typically:
     - Set **public access** for the CDN/public bucket (e.g. `prod-public` or `nestlancer-public`).
     - Keep the private bucket(s) **private** and access them via signed URLs only.

3. **Enable backup with Cloud Replication (Plan A)**
   - In Backblaze B2, go to **Replication** (or **Bucket Settings** â†’ Replication for each source bucket).
   - Create replication rules:
     - **Source:** `prod-private` â†’ **Destination:** `prod-private-backup` (same account or another B2 account/region if you prefer cross-region backup).
     - **Source:** `prod-public` â†’ **Destination:** `prod-public-backup`.
   - Replication is **free** (no egress fee); you only pay storage for the replica buckets. New uploads replicate as they arrive; existing objects can sync on a schedule (e.g. daily).

4. **Create an application key**
   - Go to **App Keys** in Backblaze.
   - Create a key with:
     - Access type: **Read and Write** (or more restrictive if you separate keys per bucket).
     - Bucket access: either **All** or restrict to the specific buckets used by this project.
   - Note down:
     - **KeyID** â†’ will be used as `B2_KEY_ID`
     - **ApplicationKey** â†’ will be used as `B2_APPLICATION_KEY`

5. **Get the B2 S3-compatible endpoint**
   - In the bucket details, find the **S3 Endpoint** (usually something like `s3.us-west-004.backblazeb2.com`).
   - Use this as `B2_ENDPOINT` (including the `https://` prefix, e.g. `https://s3.us-west-004.backblazeb2.com`).

### 3. Configure environment variables for this project

In your `.env.production` / `.env.development` files (or deployment environment), set:

- **Which provider to use**
  - `STORAGE_PROVIDER=b2`

- **B2 connection**
  - `B2_KEY_ID=<your B2 keyID>`
  - `B2_APPLICATION_KEY=<your B2 applicationKey>`
  - `B2_ENDPOINT=<your B2 S3 endpoint, e.g. https://s3.us-west-004.backblazeb2.com>`
  - `B2_REGION=us-east-1` (or the region matching your endpoint if different)

- **Canonical bucket names** (match your actual B2 bucket names)
  - Default (many buckets): use the names below.
  - **Plan A** (2 physical buckets + 2 replicas): use the same two bucket names for all logical buckets as in the table below.

  **Default (separate buckets):**
  - `STORAGE_BUCKET_PRIVATE=nestlancer-private`
  - `STORAGE_BUCKET_PUBLIC=nestlancer-public`
  - `STORAGE_BUCKET_AVATARS=nestlancer-avatars`
  - `STORAGE_BUCKET_ATTACHMENTS=nestlancer-requests`
  - `STORAGE_BUCKET_QUOTES=nestlancer-quotes-pdfs`
  - `STORAGE_BUCKET_DELIVERABLES=nestlancer-deliverables`
  - `STORAGE_BUCKET_REPORTS=nestlancer-reports`
  - `STORAGE_BUCKET_PDFS=nestlancer-pdfs`

  **Plan A (all private â†’ one bucket, all public â†’ one bucket):**
  - `STORAGE_BUCKET_PRIVATE=prod-private`
  - `STORAGE_BUCKET_PUBLIC=prod-public`
  - `STORAGE_BUCKET_AVATARS=prod-public`
  - `STORAGE_BUCKET_ATTACHMENTS=prod-private`
  - `STORAGE_BUCKET_QUOTES=prod-private`
  - `STORAGE_BUCKET_DELIVERABLES=prod-private`
  - `STORAGE_BUCKET_REPORTS=prod-private`
  - `STORAGE_BUCKET_PDFS=prod-private`

  (If avatars must be private, set `STORAGE_BUCKET_AVATARS=prod-private`.)

These values are read by `NestlancerConfigService` and the storage modules; no code change is needed if you keep or correctly override these names.

### 4. Connect Backblaze B2 to Cloudflare CDN

This assumes:

- You have a domain managed in Cloudflare (e.g. `cdn.example.com` or `assets.example.com`).
- You want Cloudflare to sit in front of your **public** B2 bucket (e.g. `nestlancer-public`) to cache and serve static files.

#### 4.1 Get your B2 bucket public URL

1. In Backblaze, open your **public** bucket (e.g. `nestlancer-public`).
2. Find the **Friendly URL** or **S3-style URL**, which looks like:
   - `https://f004.backblazeb2.com/file/nestlancer-public/...` (friendly) or  
   - `https://s3.us-west-004.backblazeb2.com/nestlancer-public/...` (S3-style)

For Cloudflare CDN, you will typically point a CNAME at the **S3-style endpoint** host (e.g. `s3.us-west-004.backblazeb2.com`) or the bucket host provided in Backblaze docs.

#### 4.2 Add CNAME in Cloudflare

1. Go to your domain in the **Cloudflare Dashboard** â†’ **DNS**.
2. Add a new DNS record:
   - Type: `CNAME`
   - Name: e.g. `cdn` (this will give you `cdn.yourdomain.com`)
   - Target: the B2 host for your bucket, for example:
     - `s3.us-west-004.backblazeb2.com` (or the exact host Backblaze shows for your bucket)
   - Proxy status: **Proxied** (orange cloud) so Cloudflare acts as CDN.
3. Save the record.

#### 4.3 Configure SSL/TLS in Cloudflare

1. In Cloudflare, go to **SSL/TLS** â†’ **Overview**.
2. Set **Encryption mode** to at least **Full** (or **Full (strict)** if you configure a valid certificate on the origin).

#### 4.4 Ensure proper caching headers from B2

Cloudflare respects cache headers from the origin. For good CDN performance with B2:

1. In Backblaze, edit your public bucket **Bucket Info** (metadata).
2. Add a JSON object with cache-control, for example:

   ```json
   {"cache-control": "max-age=86400"}
   ```

   - `max-age=86400` means Cloudflare (and browsers) can cache files for 24 hours.
   - Adjust as needed based on how often your assets change.

3. Alternatively, you can control caching via **Cloudflare Cache Rules**:
   - In Cloudflare dashboard â†’ **Caching** â†’ **Cache Rules**.
   - Create a rule matching `cdn.yourdomain.com/*` and set:
     - **Cache level**: Cache everything
     - **Edge TTL**: e.g. 1 day

### 5. (Optional) Restrict access and avoid exposing other buckets

If your CNAME points at a generic Backblaze host (like `f004.backblazeb2.com`), you should:

1. In Cloudflare, create **Page Rules** / **Transform Rules** / **URL Rewrite Rules** (depending on plan and UI):
   - Force all paths on your CDN subdomain to include only your public bucket name (e.g. rewrite `/path/file` to `/file/nestlancer-public/path/file` if needed).
   - Block/redirect any patterns that might expose other buckets.

2. Keep **private** buckets (e.g. `nestlancer-private`) configured as **private** in Backblaze and access them only through signed URLs generated by this backend.

### 6. Verify application-side configuration

Once B2 + Cloudflare are configured:

1. **Check env values**:
   - Confirm `STORAGE_PROVIDER=b2` is set in the environment used by the services.
   - Confirm all `STORAGE_BUCKET_*` names match the actual B2 bucket names.
   - Ensure `B2_KEY_ID`, `B2_APPLICATION_KEY`, `B2_ENDPOINT`, and `B2_REGION` are set.

2. **Run health checks**:
   - The storage health indicator and media worker will use the configured buckets.
   - Deploy the services and check logs to ensure uploads/reads succeed and URLs look like:
     - `https://s3.us-west-004.backblazeb2.com/<bucket>/<key>`
     - or your Cloudflare CDN hostname (if you rewrite URLs at a higher layer).

3. **Test CDN delivery**:
   - Upload a public file (e.g. from the media service using the public bucket).
   - Access it via your Cloudflare hostname, e.g. `https://cdn.yourdomain.com/path/to/file.jpg`.
   - Confirm:
     - It loads correctly.
     - Response headers in browser dev tools show `CF-Cache-Status: HIT` after the first request (meaning Cloudflare is caching).
